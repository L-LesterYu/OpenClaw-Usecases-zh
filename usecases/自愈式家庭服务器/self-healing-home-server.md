# 自愈式家庭服务器与基础设施管理

运行家庭服务器意味着你需要全天候待命。服务在凌晨3点宕机，证书悄然过期，磁盘空间耗尽，Pod 陷入崩溃循环——而你可能正在熟睡或外出。

这个用例将 OpenClaw 变成一个持久运行的基础设施运维智能体，拥有 SSH 访问权限、自动化定时任务，能够在问题发生之前检测、诊断并修复故障。

## 痛点

家庭实验室运营者和自托管用户面临着持续的运维负担：

- 健康检查、日志监控和告警需要手动配置和持续关注
- 出现故障时，你必须 SSH 登录、诊断并修复——往往是在手机上操作
- 基础设施即代码（Terraform、Ansible、Kubernetes 清单）需要定期更新
- 关于你基础设施的知识只存在于你的脑子里，而非可搜索的文档中
- 例行任务（邮件分类、部署检查、安全审计）每周消耗数小时

## 功能介绍

- **自动化健康监控**：基于 Cron 定时检查服务、部署和系统资源状态
- **自愈能力**：通过健康检查检测故障并自主修复（重启 Pod、扩缩资源、修复配置）
- **基础设施管理**：编写并应用 Terraform、Ansible 和 Kubernetes 清单
- **晨间简报**：每日推送系统健康状态、日历、天气和任务看板摘要
- **邮件分类**：扫描收件箱，标记可操作的邮件，归档噪音信息
- **知识提取**：将笔记和对话导出处理为结构化、可搜索的知识库
- **博客发布流水线**：草稿 → 生成封面 → 发布到 CMS → 部署到托管平台——全程自动化
- **安全审计**：定期扫描硬编码密钥、特权容器和过度宽松的访问权限

## 所需技能

- `ssh` 访问家庭网络中的机器
- `kubectl` 用于 Kubernetes 集群管理
- `terraform` 和 `ansible` 用于基础设施即代码
- `1password` CLI 用于密钥管理
- `gog` CLI 用于邮件访问
- 日历 API 访问权限
- Obsidian 知识库或笔记目录（用于知识库功能）
- `openclaw doctor` 用于自诊断

## 如何设置

### 1. 核心智能体配置

在 AGENTS.md 中为你的智能体命名并定义其访问范围：

```text
## Infrastructure Agent

你是一个名为 Reef 的基础设施管理智能体。

访问权限：
- SSH 到家庭网络中的所有机器（192.168.1.0/24）
- kubectl 用于 K3s 集群管理
- 1Password 密钥库（仅读取凭证，使用专用的 AI 密钥库）
- 通过 gog CLI 访问 Gmail
- 日历（你本人和伴侣的）
- Obsidian 知识库位于 ~/Documents/Obsidian/

规则：
- 永远不要硬编码密钥——始终使用 1Password CLI 或环境变量
- 永远不要直接推送到 main 分支——始终创建 PR
- 将 openclaw doctor 作为自健康检查的一部分运行
- 将所有基础设施变更记录到 ~/logs/infra-changes.md
```

### 2. 自动化定时任务系统

这套方案的核心在于定时任务调度。在 HEARTBEAT.md 中配置：

```text
## Cron 调度

每 15 分钟：
- 检查看板中正在进行中的任务 → 继续执行

每小时：
- 监控健康检查（Gatus、ArgoCD、服务端点）
- 分类处理 Gmail（标记可操作项，归档噪音）
- 检查未响应的告警或通知

每 6 小时：
- 知识库数据录入（处理新的 Obsidian 笔记）
- 自健康检查（openclaw doctor、磁盘使用率、内存、日志）

每 12 小时：
- 代码质量和文档审计
- 通过 Loki/监控栈进行日志分析

每日：
- 凌晨 4:00：夜间头脑风暴（探索笔记之间的关联）
- 上午 8:00：晨间简报（天气、日历、系统状态、任务看板）
- 凌晨 1:00：效率评估（流程改进）

每周：
- 知识库质量审查
- 基础设施安全审计
```

### 3. 安全设置（至关重要）

这部分没有商量的余地。在授予智能体 SSH 访问权限之前：

```text
## 安全检查清单

1. Pre-push 钩子：
   - 在所有仓库中安装 TruffleHog 或类似的密钥扫描器
   - 阻止任何包含硬编码 API Key、Token 或密码的提交

2. 本地优先的 Git 工作流：
   - 使用 Gitea（自托管）在推送到公共 GitHub 之前管理私有代码
   - CI 扫描流水线（Woodpecker 或类似工具）在任何公开推送之前运行
   - 合并到 main 分支前需要人工审核

3. 纵深防御：
   - 为 AI 智能体使用专用的 1Password 密钥库（限制范围）
   - 对敏感服务进行网络分段
   - 每日自动安全审计，检查：
     * 特权容器
     * 代码或配置中的硬编码密钥
     * 过度宽松的文件/网络访问权限
     * 已部署镜像中的已知漏洞

4. 智能体约束：
   - 分支保护：main 分支需要 PR，智能体不能绕过
   - 非必要场景使用只读访问
   - 所有变更通过 git 记录并可审计
```

### 4. 晨间简报模板

```text
## 每日简报格式

在上午 8:00 生成并发送：

### 天气
- [你的位置] 的当前天气和预报

### 日历
- 你今天的日程
- 伴侣今天的日程
- 标记冲突或重叠

### 系统健康
- 所有机器的 CPU / 内存 / 存储状态
- 服务：上线/离线状态
- 最近部署（ArgoCD）
- 最近 24 小时内的告警

### 任务看板
- 昨日完成的卡片
- 进行中的卡片
- 需要关注的阻塞项

### 亮点
- 夜间头脑风暴中的值得关注的发现
- 需要处理的邮件
- 本周即将到来的截止日期
```

## 关键洞察

- **"不敢相信我现在有一个能自我修复的服务器了"**：智能体可以运行 SSH、Terraform、Ansible 和 kubectl 命令来修复基础设施问题，在你发现问题之前就已经搞定
- **AI 会硬编码密钥**：这是头号安全风险。如果你不设置防护措施，智能体会毫不犹豫地把 API Key 写进代码里。Pre-push 钩子和密钥扫描是强制性的
- **本地优先的 Git 工作流必不可少**：永远不要让智能体直接推送到公开仓库。使用私有 Gitea 实例作为带 CI 扫描的暂存区
- **定时任务才是真正的产品**：调度自动化（健康检查、邮件分类、简报）提供的日常价值远超临时命令
- **知识提取的复利效应**：将笔记、对话导出和邮件处理为结构化知识库，其价值会随时间不断增长——有用户仅从 ChatGPT 历史记录中就提取了 49,079 条原子化知识

## 灵感来源

这个用例基于 Nathan 的详细文章《[Everything I've Done with OpenClaw (So Far)](https://madebynathan.com/2026/02/03/everything-ive-done-with-openclaw-so-far/)》，他在文中描述了自己的 OpenClaw 智能体 "Reef"——运行在家庭服务器上，拥有所有机器的 SSH 访问权限、一个 Kubernetes 集群、1Password 集成以及包含 5,000+ 笔记的 Obsidian 知识库。Reef 运行着 15 个活跃的定时任务、24 个自定义脚本，并已自主构建和部署了包括任务管理 UI 在内的应用程序。Nathan 在第一天 API Key 泄露后总结的血泪教训："AI 助手会毫不犹豫地硬编码密钥。它们有时缺乏人类的直觉。"他的纵深防御安全方案（TruffleHog pre-push 钩子、本地 Gitea、CI 扫描、每日审计）是任何尝试此模式的必读材料。

同时参考了 [OpenClaw Showcase](https://openclaw.ai/showcase) 上 `@georgedagg_` 的描述：部署监控、日志审查、配置修复和 PR 提交——遛狗的功夫就全搞定了。
